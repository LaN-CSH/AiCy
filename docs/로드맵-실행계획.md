## 1. 개발 트랙 — 우선순위별 작업 목록

### P1-A. 발음 정규화 (한국어 TTS의 영어/약자 문제) — 난이도 하
- **문제**: LANG=ko에서 "GPU"를 "찌뿌"처럼 읽음. 영어 단어/약자 발음 전반이 불안.
- **구현**:
  1. `backend/normalize.py` 신설 — TTS 직전 텍스트 변환 (파이프라인의 `tts.synthesize` 호출 전).
  2. 1차: 약자 사전 치환 (`GPU→지피유, AI→에이아이, API→에이피아이, TTS→티티에스, LLM→엘엘엠, OBS→오비에스, PC→피시, URL→유알엘` 등) + 정규식으로 미등록 대문자 약자(2~5자)는 알파벳명 나열(`X→엑스` 테이블)로 자동 음차.
  3. 2차(선택): 페르소나에 "영어 단어는 한글 발음으로 표기해서 말해" 지시 추가 — LLM이 원천에서 처리.
- **완료 기준**: 채팅으로 "GPU 어때?" → "지피유"로 들림. 사전은 `.env` 아닌 py 파일에 (수정 쉬움).
- **주의**: 자막(full 텍스트)은 원문 유지, TTS 입력만 변환할지 결정 필요 → 자막 원문 유지 권장.

### P1-B. 감정→표정 매핑 + 소품 토글 체계 — 난이도 중
- **현황**: 백엔드가 `emotion`을 이미 보냄. xl 모델 exp3 7개(=Param41/43/46/47/57/Param/Param2, 값 30)는 **표정인지 소품 토글인지 미확인**.
- **구현**:
  1. 사용자와 함께 각 파라미터 정체 확인: 컨트롤 창 콘솔에서 `aicyParam("Param41",30)` 등 하나씩 → 무엇이 바뀌는지 기록 (5분 작업).
  2. `frontend`에 레지스트리: `{happy: [["ParamXX",30]], sad: ..., props: {sword: ...}}` — 감정 도착 시 해당 파라미터 세트 적용, 발화 끝나면 해제(또는 유지 정책).
  3. 소품(무기 등)은 별도 채널: LLM 출력에 `[prop:sword]` 태그 규약 추가 → `persona.extract_emotion` 옆에 `extract_props` → WS로 전달 → 프론트 forcedParams 적용. 페르소나에 사용 규칙 명시("칼싸움 얘기할 때만" 등).
  4. 참고: Open-LLM-VTuber(MIT)의 emotionMap 구현 — 모델별 감정→표정 인덱스 방식.
- **완료 기준**: `[happy]` 답변에서 표정 변화가 눈에 보임. 소품 1개 이상 태그로 on/off.

### P1-C. 대화 기록 DB화 → compact → RAG — 난이도 중 (단계별)
- **문제**: conversation 무한 누적(토큰 비용↑) + 재시작 시 전부 소실.
- **구현 (순서대로, 각각 독립 배포 가능)**:
  1. **로깅**: SQLite(`chat_history.db`, gitignore) — 모든 (시각, source, author, user_text, reply, emotion) 기록. `brain.respond` 훅. *쉽고 즉시 가치(데이터 축적 = 나중에 페르소나 파인튜닝 재료).*
  2. **auto-compact**: conversation이 N턴(예: 30) 넘으면 오래된 절반을 LLM으로 요약 → system 메시지 1개로 치환. 방송 중 토큰 폭주 방지.
  3. **RAG**: 답변 전 현재 채팅과 유사한 과거 기록 k개 검색(임베딩: OpenAI text-embedding-3-small 또는 로컬 bge-m3) → 컨텍스트 주입. "지난번에 OO 얘기했잖아"가 가능해짐.
- **완료 기준**: 1) 재시작 후에도 DB에 기록 존재 2) 장시간 대화에서 토큰 상한 유지 3) 과거 대화 회상 답변 확인.

### P2-A. 방송 디렉터 (idle 혼잣말 + 채팅 반응 비율) — 난이도 중상
- **사용자 요구**: "컨텐츠에 대한 혼잣말도 해야 하고, 채팅 반응이 방송의 대부분이 될 필요는 없다."
- **설계**: orchestrator에 `director` 루프 —
  1. **idle 혼잣말**: 마지막 발화 후 X초(45~90 랜덤) 무채팅이면 혼잣말 생성. 토픽 소스: `topics.txt`(사용자가 채울 콘텐츠 큐) → 소진 시 잡담 시드(오늘 상태, 시청자 질문 유도 등).
  2. **반응 정책**: 콘텐츠 진행 중 채팅은 즉시 응답하지 않고 큐에 쌓음 → 문단 사이에 배치 반응("아 채팅 보니까 OO님이..."). MVP는 단순 우선순위(채팅 오면 idle 취소)로 시작, 정책은 점진 고도화.
  3. 페르소나에 혼잣말 모드 지시 추가(질문형 마무리 금지 등, 채팅 유도형).
- **완료 기준**: 채팅 없이 10분 방치해도 방송이 침묵하지 않음. 채팅 오면 자연스럽게 전환.

### P2-B. 안전필터 실장 — 난이도 중
- **현황**: `safety.check_input`이 무조건 통과. 공개 방송에는 부족.
- **구현**: 1) 입력: 금칙어/도배/멘션폭탄 필터 + 무시 정책(조용히 스킵) 2) 출력: OpenAI moderation API(무료)로 답변 검사 → 걸리면 회피 멘트로 교체 3) 페르소나 안전선과 이중화.
- **완료 기준**: 욕설/유도 채팅에 안전하게 무반응 or 회피. 필터 로그 남김.

### P2-C. 장시간 안정 운영 — 난이도 중
- YouTube API 쿼터: 현 5초 폴링 ≈ 시간당 3.6k유닛(일 10k 한도) → **3시간이 한계**. 대응: 폴링 간격 동적 조절(채팅 빈도 따라 5~20초) / pytchat류 비공식 수집기 검토(쿼터 무관, ToS 회색지대 명시).
- 재연결: WS 끊김·YouTube 폴링 실패·TTS 예외 시 자동 복구 (일부 있음, 전수 점검).
- 방송 시작 자동화: `/yt` 대신 채널 ID로 현재 라이브 자동 탐지(`search.list`는 100유닛 — videos.list 쓰는 방법 강구).
- **완료 기준**: 6시간 무인 방송에서 채팅 반응 지속.

### P3-A. 채팅 UI 다듬기 — 난이도 하
- 말풍선/라이브 패널 디자인 개선(닉네임 색, 등장 애니메이션, 이모지, 슈퍼챗 대비 스타일), 방송 오버레이 가독성(폰트 크기·그림자). 사용자 취향 확인 필수 — 시안 2~3개 비교 제안.

### P3-B. 목소리 품질 개선 — 난이도 중 (연구 트랙과 연결)
- **사용자 평가**: "나쁘진 않은데 좀 바보 같고, 말끝 올라갈 때 가끔 깨짐."
- **원인 가설**: ① ref 클립이 합성음(SunHi) 기반 → 클로닝 상한 낮음 ② chatterbox 한국어는 다국어 모델이라 영어 대비 약함 ③ EOS 부근 아티팩트(말끝 깨짐은 chatterbox 알려진 특성).
- **개선 순서**: 1) **ref 클립 교체** — 진짜 사람 음성(권리 확보) 10~15초로 오디션(가장 효과 큼, 사용자 액션) 2) `AICY_CB_CFG`(0.3~0.7)·`AICY_CB_EXAGGERATION` 튜닝 + 문장 끝 구두점 정리 3) 근본 해결은 연구 트랙 B/C단계(파인튜닝).

## 2. 연구 트랙 (배경: `docs/TTS-연구지도.md`)

| 단계 | 내용 | 상태 |
|---|---|---|
| A. 무학습 통합 | Chatterbox/Turbo 로컬 실측·백엔드 통합 | ✅ 완료 (RTF: ko 1.21, turbo 0.4~0.6) |
| B. 보이스 정체성 | AiCy 전용 목소리 — 클로닝 고도화 or LoRA(SLURM) | ⬜ 다음 |
| C. 표현 파인튜닝 | 비언어(웃음 등) 태깅 데이터로 LoRA → 4060 Ti 배포 | ⬜ |
| D. 연구급 | 음성 DPO, 대화 문맥 운율(이전 턴 조건화) | ⬜ |

### B단계 구체 계획 (다음 연구 작업)
1. Chatterbox 파인튜닝 가능성 조사: 공식 학습 코드 유무 확인 → 없으면 대안(GPT-SoVITS 클로닝: 한국어 강함, 수분 데이터로 전용 보이스).
2. SLURM 환경 준비: 컨테이너/콘다 재현 스크립트, 데이터 업로드 경로. (사용자의 SLURM 서버 사양 확인 필요)
3. 목표 보이스 데이터 확보(사용자): 원하는 톤의 음성 10분~1시간(권리 확보).

### 연구 아이디어: 성대모사(vocal imitation) — 사용자 핵심 관심사
- **정의**: speech가 아닌 소리(사이렌·동물·효과음)를 들려주면 **사람 목소리로 흉내내는** 오디오 생성. 버튜버 개인기·밈 요소.
- **접근 로드맵**:
  1. (즉시 가능한 근사) 페르소나가 의성어 텍스트로 흉내 + exaggeration↑ — 개그 효과용.
  2. (C단계 연계) 사람이 효과음을 흉내낸 녹음 데이터 수집(직접 녹음/크라우드) + `<imitate:siren>` 태그 학습 → 태그로 성대모사 시전.
  3. (연구) audio-prompted generation: 비음성 오디오를 조건으로 넣어 사람 음색으로 재해석 — Bark류 토큰 LM의 오디오 프롬프팅, MIT 조음모델 기반 vocal imitation 연구 서베이부터.
- **첫 실험 제안**: Turbo 태그 체계에 커스텀 태그가 추가 학습 가능한지 확인하는 소규모 LoRA (SLURM).

## 3. 실행 방식 (다음 세션/모델을 위한 프로토콜)

2. 자동 메모리(`MEMORY.md` — 프로젝트 폴더 귀속이라 모델 무관 유지됨)에 최근 상태 있음.

### 작업 원칙 (이 프로젝트에서 검증된 방식)
- **실측 우선**: 코드 작성 → `_smoke_*.py`로 실제 실행 검증(가짜 Brain + sapi/gtts로 키·GPU 없이) → 삭제 → 커밋. "될 것이다"로 커밋하지 않는다.
- **작게 자주 커밋**: 기능 단위, 한국어 메시지, main 직push.
- **사용자 확인이 필요한 것**: 페르소나 톤, 목소리 선택, UI 디자인, 방송 정책(반응 비율), 유료 API 사용, 외부 서비스 가입.
- **환경 함정은 CLAUDE.md 참조** (PowerShell·.venv311·OBS 캐시·cp949 등 전부 기록돼 있음).
